# HCG 제안서 기획(Planning) 가이드라인 v1.0

> **계승:** 기존 hcg-ppt-design skill md **v1.0**의 핵심원칙·논리구조·헤드메시지·콘텐츠 작성·Do&Don't 영역.
> **분석 기반:** 10개 제안서 약 530개 파일(기아·KB라이프·NH투자증권·크래프톤·파라다이스·하이트진로·한화손보·현대차·리파인·롯데알미늄) — fb 피드백 텍스트 4 + fb PPTX 8 + 내부회의록 20+.
> **보강:** 롯데알미늄 _final(25장)·크래프톤 OKR v2.0(79장)·한림제약_fin·코웨이_final·매일그룹 final 구조 재분석.

> **JSON 버전:** `skill_ppt_planning.json` (기계 판독/프롬프트용). 본 .md는 사람 판독/리뷰용. 두 버전은 동일 내용을 별도 관리.

## 0. 최우선 규칙 (Priority Rules)

| # | 규칙 |
|---|------|
| **RULE-1** | `fb_` 피드백 + `_final` 최종본 기준이 초안과 충돌 시 **항상 fb_/_final 우선**. |
| **RULE-2** | 핵심은 '제도'가 아니라 **'연결 구조'**. 제도(MBO/OKR/직무급)는 껍질, 본질은 평가-보상-승진 연계(sub-infra). |
| **RULE-3** | **고객 민감도 우선**. 첫 장부터 강한 진단('하드모드') 금지. 배경/Pain Point는 RFP 근거 내 절제(게임사·노조 특히). |
| **RULE-4** | 가설 단정 금지 → **'도출 프로세스/순서'**를 보여줄 것(가설 아웃풋보다 프로세스 맵). |
| **RULE-5** | 분량은 고객 요청 최우선. 산출물 과다 = **중압감·패키지 딜 인상**. 중복 장표 통합. |

## 1. 핵심 원칙 (Core Principles)

| # | 원칙 | 근거 (실제 피드백) |
|---|------|----------------------|
| **P1** | **인사이트가 없으면 슬라이드가 아니다** | "너무 뻔한데 좀 업그레이드(형식+내용)" (한화 fb) |
| **P2** | **고객사 문맥 없는 방법론은 공허하다** | "우리 인사이트를 여기에 녹이는 게 전략" (기아 fb) |
| **P3** | **슬라이드는 주장이다, 보고서가 아니다** | "현황은 시사점만. 헤드 메시지부터 정리 후 현황" |

## 2. 핵심 메시지 도출 (Core Message Doctrine)

**원칙:** 제안서 전체가 단 하나의 핵심 주장으로 수렴. 모든 장표는 그 주장의 backup.

**도출 공식:** `[고객 표면 요구] 뒤 [진짜 문제] 재정의 → [경쟁/Legacy 한계] 대비 [HCG 차별 해법]을 한 문장으로`

| 사례 | 표면 → 재정의 → 메시지 |
|------|------------------------|
| 크래프톤(OKR) | OKR 도입 → 제도는 껍질, 본질은 평가/보상 연계 → "유연한 평가·보상 연계 설계 + 리더 준비를 HCG가 가장 잘한다" |
| 롯데알미늄(직무HR) | 직무체계 수립 → Legacy 컨설팅은 자원·시간 과다 → "글로벌 표준 설계도 + AI로 직무기반 HR을 쉽고·정확하고·빠르게" |
| 코웨이/한림(성과관리) | — → 제도+코칭+시스템 분절 → "제도 설계 + 코칭 + 자체 시스템(Performance Plus) End-to-End" |

**안티패턴:** MBO vs OKR 형식 비교에 메시지 낭비 금지(→ '다를 게 없다, 본질은 연계'로 빠르게 전환) / 방법론 자체로 차별화 금지(베끼기 쉬움) / 교육 필요성 과잉 강조 금지.

### 2-1. 헤드 메시지 작성 원칙

| 원칙 | Do | Don't |
|------|-----|-------|
| **결론 선행** | "HR ROI 저하는 보상-성과 연계 부재의 구조적 결과임" | "한화손보 현황 분석" |
| **중립성 유지** | "전략 연계형 인력운영 전환이 중장기 경쟁력의 핵심임" | "문제가 심각하므로 개선이 시급함" |
| **전체 통합** | 하단 2~3개 논거를 아우르는 하나의 명제 | 세부 항목 하나만 반영한 좁은 헤드 |
| **오해 방지** | 앞 내용과 연결해 특정 요소만 지칭하는 오해 차단 | "보상수준 = BM 2개만 활용" 식 오해 유발 |

## 3. 스토리라인 / 논리 구조 (Story Architecture)

### 3-1. 표준 목차 흐름 (대챕터 3~4개만, 세부목차 X)

```
[Cover]  프로젝트 정식 명칭 + "- 제안서 -" + 제안 연월   ※ 강한 진단 카피 금지
[목차]   로마자 Ⅰ~Ⅳ 대챕터, active/inactive, 저작권 문구 하단 고정
  ├── Ⅰ. 프로젝트 배경 / Overview
  │     Project Overview(내용/프로세스/방안) → 고객 이해·내부 Issue
  │     → Client's Needs & Pain Point(As-Is) → HCG's Approach(To-Be, 1:1)
  │     → 문제 본질 재정의(제도가 아니라 연계) → 선진/실패 사례(MS·Google·Adobe / 두산·포스코)
  │     → Ideal Model & 변화 전략 → Why HCG(경험 / A-to-Z 코칭 / 자체 시스템)
  ├── Ⅱ. 추진방안 (방법론)
  │     Overall Approach(3모듈+변화관리 동반)
  │     → 모듈1 현황진단·제도설계(진단→이슈정의→A 평가→B 보상)
  │     → 모듈2 Implementation(교육/코칭/변화관리 Roadmap)
  │     → 모듈3 시스템 Implementation(솔루션/연동/업데이트)
  ├── Ⅲ. 일정 및 조직
  │     전체 일정(간트, 최소 3개월) → 수행 조직 → 투입 인력 Profile
  └── Ⅳ. HCG 소개
        Overview → 사업 영역 → 컨설팅 서비스 영역 → 주요 고객사(870여 곳) → Long-term Relationship
[End]
```

### 3-2. 슬라이드 단위 논리 (1 Slide = 1 Message)

```
┌──────────────────────────────────────────────┐
│ [헤드 메시지]  결론을 담은 완결 문장 (1~2줄)     │
├──────────────────────────────────────────────┤
│ 본문: 3/4분법 도해 또는 프로세스 흐름            │
│       팩트/데이터 → 해석 → 시사점 흐름           │
├──────────────────────────────────────────────┤
│ [선택] 각주: 데이터 출처 / 기준 시점             │
└──────────────────────────────────────────────┘
```

### 3-3. 배경 슬라이드 전개 패턴

- **틀린 흐름 (Don't):** `문제 나열 → 솔루션 제시`
- **올바른 흐름 (Do):** `변화 상황 묘사 → 현상(징후/파열음) → 가설적 이슈 → HCG 관점 핵심 원인 → 대응 방향`
- 첫 슬라이드: 산업 현황 → 고객사 특유 현황. 매출/이익/사건 등 팩트 기반.
- implication은 별도 강조 없이 헤드 메시지에 녹임. 세부 데이터는 `[참고]`로 분리.

### 3-4. 장표 논리 규칙

- 각 장표는 앞 장표 결론에서 자연 연결(50장 이슈 → 51장 그 포인트/포커스).
- 한 장표 = 한 메시지. 제목 = 메시지 자체이거나 그 카테고리.
- 제목 카테고리/섹션 prefix 전 장표 일관(예: `Ⅰ. 프로젝트 배경  소제목`).
- 비교 장표는 '차이 없음 → 그래서 본질은 연계' 결론으로 빠르게 전환.
- 중복/유사 장표 통합. `[참고]/[Back-up]`은 본문 흐름에서 분리.
- 좌=개요/방법, 우=아웃풋/결과 이미지 좌우 내러티브를 모듈 전반 통일.
- 단정 불가 내용(시스템 연동 등)은 '탐색/추진하겠다' 가능성 언어. 로고 무단 사용 금지.

## 4. 콘텐츠 섹션별 작성 원칙

### 4-1. Project Overview
3행 — **추진 내용 / 추진 프로세스(4단계 박스+화살표) / 추진 방안**. 추진 방안 필수: 기간(예 12주) · 인력(예 PM 1+컨설턴트 2) · 금액(예 225,000천원, 부가세 별도, 협의 변동).

### 4-2. Pain Point ↔ Approach
As-Is(문제) ↔ To-Be(해법) **2컬럼 1:1 대응**. 행마다 짝 일치. 톤은 사실 기반 절제, 고객 비난 인상 금지.

### 4-3. Why HCG / 차별성
일반 접근(기본 요구수준) vs HCG(가치 창출) 명확 대비, 경쟁대안=회색. **강점 3대 축: ① 유관 프로젝트 경험(레퍼런스 구체화) ② 제도수립~코칭 A-to-Z ③ 자체 솔루션/시스템 보유.** 유사 사례는 `[참고]/[Back-up]`.

### 4-4. 수행 방안 (방법론)
4단계 표준(Modeling → 실시 → 분석 → 개선방안). 각 단계 핵심 산출물만(과도 나열 삭제). 단계마다 유사 프로젝트 경험 1~2줄. 아웃풋 먼저 → '다년 매뉴얼로 체계적·신속'으로 받기.

### 4-5. 레퍼런스 / 수행사례
고객사 산업·유사 과업 기준 선정(무관 사례 금지). 서술형(고객사명 + 과업 한 줄 + 수행 요약). 그룹사는 별도 묶기. 오래된/부적합 연도 제거.

**Backbone:** 성과관리/OKR(기아·코오롱·GS·녹십자·코웨이·한림·네오이뮨텍·코스맥스), 직무HR(롯데알미늄·코오롱인더·포스코인터), 시스템(매일그룹·미래에셋). 모듈 성격(제도설계/코칭/시스템)에 맞춰 적재적소 배치.

## 5. 리더 피드백 Playbook (fb_/내부회의 패턴)

- **스토리 흐름 점검:** '어떤 흐름으로 장표가 배치되었는가'에 한 문장 답 가능해야.
- **메시지 손질:** 핵심 장표 메시지가 본질(연계)을 가리키는지 재확인.
- **배경 절제:** 빈약해도 무리한 추정 진단 금지. 고객 정서 리스크 우선.
- **산출물 통제:** 많아 보이면 축소. 맥락상 의미 없는 산출물 삭제(중압감 회피).
- **패키지 딜 회피:** 자사 시스템 전제 금지, '타 시스템도 가능' 열기.
- **차별화 위치:** Why HCG '시스템 보유'는 실패사례/상시관리 맹점 뒤에 붙이면 설득력↑.
- **거버넌스:** 제출 전 부사장/임원 리뷰 → 컨펌 → 제출.

## 6. Do & Don't 총람 (내용/논리)

| 구분 | Do ✅ | Don't ❌ |
|------|--------|----------|
| 헤드 메시지 | 결론/인사이트 담은 완결 문장 | 슬라이드 제목 수준 주제어 |
| 배경 전개 | 변화 → 현상 → 가설 이슈 흐름 | 문제 직접 나열 후 솔루션 |
| 범위 설정 | 프로젝트 범위 내 구체 내용 | 비전/전략/가치 등 범위 초과 일반론 |
| 인사이트 | HCG 고유 관점 언어 | "~가 필요합니다" 류 뻔한 결론 |
| 데이터 | 출처/시점 명시 구체 수치 | 출처 없는 임의 통계 |
| 산출물 | 핵심 3~5개만 | 전체 목록식 나열 |
| 레퍼런스 | 고객 유관 서술형 | 관련성 낮은 로고 나열 |
| 목차 | 대챕터 3~4개 | 세부 항목까지 나열 |

## 9. 기획 품질 체크리스트

- [ ] 핵심 메시지가 한 문장으로 존재하고 모든 섹션이 backup하는가?
- [ ] `fb_`/`_final` 기준과 충돌하는 배치가 없는가? (RULE-1)
- [ ] 문제 본질(연계 구조)이 제도 형식 비교에 묻히지 않았는가? (RULE-2)
- [ ] 각 헤드가 해당 장 결론/인사이트를 담는가? '뻔한 결론'·일반론 장표는 없는가? (P1)
- [ ] 고객사 특유 맥락(산업/전략/데이터)이 방법론에 연결됐는가? (P2)
- [ ] 배경이 변화→현상→가설→HCG관점→대응 흐름인가?
- [ ] 고객 민감 포인트(첫 장 톤, 노조/세대, 패키지 딜)를 회피했는가? (RULE-3)
- [ ] 장표 간 인과가 끊긴 곳이 없고 중복 장표를 통합했는가?
- [ ] 제목 카테고리/섹션 prefix가 일관되는가?
- [ ] 산출물/일정/비용이 중압감 없이 고객 요청·현실성에 부합하는가?
- [ ] 레퍼런스가 고객 유관 사례로 선정·서술형인가?
- [ ] Why HCG 3대 축(경험/A-to-Z/시스템)이 모두 있는가?
